#RULE_INFRACTIONS STICKY — SETUP
===============================

WHAT THIS DOES
--------------
Keeps the sticky message pinned to the BOTTOM of #rule_infractions, even when
the bot auto-posts infractions into it.

THE PROBLEM IT FIXES
--------------------
A plain sticky re-posts itself on a Regex `.*` trigger, which fires on every new
message in the channel. That worked while staff posted infractions BY HAND —
each human message re-fired the trigger and the sticky dropped back to the
bottom.

The new auto-infraction commands post the pings as the BOT. YAGPDB custom-command
triggers never fire on the bot's own messages, so the `.*` trigger stops firing
and the sticky gets stranded ABOVE the bot's pings.

This version still runs on the normal `.*` trigger for human posts, AND is called
directly (via execCC) by the auto-infraction senders right after they post a
ping — so it re-sticks underneath the bot's messages too.


BEFORE YOU START
----------------
1. Turn on Discord Developer Mode:
   User Settings → Developer → Developer Mode (ON).

2. Copy this channel ID (right-click channel → Copy Channel ID):
   • #rule_infractions


THE EASIEST PATH — EDIT YOUR EXISTING STICKY
--------------------------------------------
You almost certainly already have a sticky command on #rule_infractions. You do
NOT need to recreate it or retype its text — just make three small edits to it:

  1. At the very top of the code, ABOVE the line that builds your embed/message,
     add these two lines:

         {{ $ch := .Channel.ID }}
         {{ if .ExecData }}{{ $ch = .ExecData.stickyChannel }}{{ end }}

  2. In the "do not edit below" block, replace every `nil` channel argument with
     `$ch`:

         {{ if $db := dbGet $ch "stickymessage" }}
             {{ deleteMessage $ch (toInt $db.Value) 0 }}
         {{ end }}
         {{ $id := sendMessageRetID $ch $message }}
         {{ dbSet $ch "stickymessage" (str $id) }}

  3. Leave your trigger as Regex `.*` restricted to #rule_infractions, and leave
     your sticky text exactly as it is.

infractions_sticky.go in this folder is the same thing as a full reference — the
only line that is "yours" is the $message line; everything else is identical for
every server.


THEN — GET THE COMMAND ID
-------------------------
Open/edit the sticky command in the YAGPDB dashboard and look at the page URL —
it ends in a number, e.g. .../customcommands/commands/1234/ → the ID is 1234.

Write this number down. You will paste it into TWO other commands:
  • Alert Sender   (alert_sender.go)   → {{ $infractionsSticky := 0 }}
  • Reaction Check (reaction_check.go) → {{ $infractionsSticky := 0 }}

That is what makes those commands re-stick the sticky after they post a ping.
See their own setup files (alert_sender, quick_reactions) for where to paste it.


INFRACTION COUNTER (3rd / 4th-strike escalation)
------------------------------------------------
This command also counts each person's infractions over a rolling 6-month window
and escalates:
  • 3rd infraction in 6 months → the infraction notice gains a warning line.
  • 4th infraction in 6 months (and EVERY infraction after) → the person is
    suspended from posting adverts for 14 days (a self-expiring `advertBan` flag
    the advert commands check) and a heads-up is posted to the bot-spam channel so
    staff know a ban landed. History is NOT wiped — entries only drop once they age
    past the 6-month window — so a member who infracts again after a ban expires is
    immediately re-muted for another 14 days.

Set the config near the top of the command:
       {{ $botSpam := 0 }}   → your bot-spam channel ID (0 disables the heads-up)
Leave $infrWindowSecs (180 days) and $advertBanSecs (14 days) unless you want to
change those durations.

The count is stored as ONE database entry per user — a list of infraction
records under the key `infractionLog` (each record holds the time, a
comma-joined reason, and the offending post's channel/message id for a jump
link), pruned to the last 6 months on every write. Older installs used a bare
timestamp list under `infractionDates`; those entries are still read and are
migrated into `infractionLog` the next time the member infracts. The bot's own
pings never fire this `.*` trigger, so the advert commands
count those; THIS branch only counts infractions a human types here by hand
(matched by the first user @mention in the message). Set the SAME $botSpam,
$infrWindowSecs, and $advertBanSecs in the three advert commands and reaction
check so every path counts identically.


EXPIRY LEDGER (feeds the 180-day channel cleanup)
-------------------------------------------------
Because this command runs after EVERY message in #rule_infractions (`.*` for
human posts, execCC for bot pings), it also records each message's ID for the
post-expiry system (see the post_expiry folder). IDs are appended to one
30-day bucket entry (`infrLedger_<n>`, owned by the channel ID) — about 7
entries exist in total at any time, regardless of infraction volume.

Nothing to configure here: if you paste the current infractions_sticky.go (or
re-apply your three "easiest path" edits on top of it), the ledger lines come
with it. The actual deleting is done by the post_expiry/infraction_expiry.go
interval command — without it the ledger just sits there, tiny and harmless.
If you edited your own sticky rather than pasting this file, copy across the
`$ledger` lines: the cslice at the top, the two branch appends, the two
sendMessageRetID escalation lines, and the flush block at the bottom.


NOTE ON RAPID-FIRE PINGS
------------------------
If several infractions land in the same second, each one nudges the sticky, and —
exactly like the original sticky on rapid human posts — there is a small race
window where a stray copy can be left behind. This matches the existing sticky
behaviour and is rare at normal infraction volume. If it ever becomes a problem,
apply the same race-guard you use on the other channel stickies.
